# Decision Log — Mealie Android PRD

## Session: 2026-05-21 (Initial PRD creation)

### DL-1: V1 includes online recipe browsing
**Decision:** V1 includes recipe list + detail view (online-only). Offline recipe caching deferred to v2.
**Rationale:** Confirmed by Xexanos. Brief's executive summary was aspirational; scope table in brief already separated offline recipe browsing as post-v1. Online browsing adds value without requiring local caching infrastructure.

### DL-2: Single server per install (v1)
**Decision:** One Mealie Instance configured per app install. Multi-server profile switching is post-v1.
**Rationale:** Confirmed by Xexanos. Simplifies setup flow and Local Store schema. Flagged multi-server as an interesting future option.

### DL-3: API 26 (Android 8.0) minimum
**Decision:** Minimum Android API 26.
**Rationale:** Confirmed by Xexanos. Chosen as a pragmatic floor for Jetpack Compose + EncryptedSharedPreferences. Revisit if a specific v2 feature requires a higher floor.

### DL-4: JWT authentication only (no API tokens)
**Decision:** Authentication uses username/password → Access Token + Refresh Token flow (`POST /api/auth/token`, `GET /api/auth/refresh`). API tokens are explicitly excluded.
**Rationale:** API tokens require Mealie admin privilege; non-admin Passengers cannot create them. JWT flow works for all user types. Documented in brief addendum.

### DL-5: Kotlin + Jetpack Compose (native Android)
**Decision:** Native Android, Kotlin + Jetpack Compose.
**Rationale:** Flutter ruled out (no Mac for iOS compilation). React Native ruled out (offline-first and Android-specific APIs better served natively). Documented in brief addendum.

### DL-6: Target Mealie API v3
**Decision:** PRD targets Mealie v3 API (v3.16.0 confirmed on Xexanos's instance).
**Rationale:** Xexanos provided the OpenAPI spec from his running instance. No explicit API versioning in path structure; endpoints use `/api/` prefix.

### DL-7: No analytics or crash reporting SDK
**Decision:** No third-party analytics or crash reporting included in the app.
**Rationale:** Consistent with open-source, privacy-respecting positioning. Recorded as an assumption in PRD §9.

### DL-8: WorkManager CONNECTED (not UNMETERED) for sync
**Decision:** Sync proceeds on mobile data connections, not restricted to WiFi/unmetered.
**Rationale:** Core use case (supermarket shopping) may occur away from home WiFi. Flagged as assumption; user can revisit if battery or data concerns arise.

### DL-9: Distribution - direct APK for v1
**Decision:** V1 distribution is direct APK. Play Store and F-Droid are post-v1 options.
**Rationale:** Inferred from project scope (small initial audience, personal/community project). Open Question 5 tracks the formal decision.

### DL-10: HTTP URLs accepted with security warning
**Decision:** HTTP and HTTPS URLs both accepted. If the user enters an HTTP URL, a security warning is shown before credentials are submitted (FR-2).
**Rationale:** Operators on local networks commonly run Mealie without TLS. Warning preserves user choice while surfacing the risk.

### DL-11: Recipe list endpoint confirmed - GET /api/recipes
**Decision:** `GET /api/recipes` confirmed via OpenAPI spec (v3.16.0). Supports `queryFilter` search and `page`/`perPage` pagination. Assumption A2 removed from PRD.
**Rationale:** Xexanos provided live OpenAPI spec from his Mealie instance; endpoint verified directly.

### DL-12: Shopping List roster pagination added
**Decision:** Pagination UI added to Shopping List roster (FR-10) regardless of typical household size.
**Rationale:** Xexanos confirmed: small implementation cost, correct behaviour regardless of list count. Assumption A3 removed.

### DL-13: Sync Network Mode setting added (FR-18)
**Decision:** New user-configurable setting: All Networks (default, CONNECTED constraint) or Wi-Fi Only (UNMETERED constraint). WorkManager job constraint reflects active mode.
**Rationale:** Users with home-only (non-public) Mealie instances need Wi-Fi Only to avoid pointless sync attempts on mobile data when the server is unreachable. Default remains All Networks to serve the supermarket use case.

### DL-14: No analytics SDK - hard constraint
**Decision:** No analytics, telemetry, or automatic crash-reporting SDK included. Promoted from assumption to hard constraint in Cross-Cutting NFRs and §5 Non-Goals.
**Rationale:** Xexanos confirmed. Open-source, privacy-respecting positioning.

### DL-15: Bug reporting - GitHub Issues link (v1), crash capture (post-v1)
**Decision:** FR-19 (v1): Settings > "Report a Bug" opens GitHub Issues new-issue URL pre-filled with device context (Android version, app version, device model). No SDK required. Post-v1: local crash capture with share-sheet opt-in (Option B) deferred to §6.2.
**Rationale:** Option A is zero-dependency and appropriate for a small, technically-inclined v1 audience. Option B adds value once the app reaches a broader user base via F-Droid / Play Store.

### DL-16: EncryptedSharedPreferences replaced with DataStore + Android Keystore
**Decision:** All references to EncryptedSharedPreferences replaced with "DataStore backed by Android Keystore encryption" throughout the PRD (Glossary, FR-3, FR-6 NFR, FR-4 NFR, Cross-Cutting NFRs, Platform section).
**Rationale:** EncryptedSharedPreferences (androidx.security:security-crypto) is deprecated. DataStore is the modern SharedPreferences replacement; combined with Android Keystore encryption it provides equivalent security with a non-deprecated API.

---

### DL-18: Connectivity probe replaces SSID-based sync filtering
**Decision:** The sync WorkManager job performs a lightweight probe against the Mealie Instance before syncing. No SSID-based filtering; no `ACCESS_FINE_LOCATION` permission requested.
**Rationale:** Reading the connected SSID requires `ACCESS_FINE_LOCATION` on Android 10+ - an alarming permission ask for a recipe app. The probe achieves the same practical effect (sync only when server is reachable) without extra permissions, and also handles VPN and future public-URL scenarios that SSID filtering cannot. The Wi-Fi Only toggle in FR-18 is retained separately for users who want to restrict sync traffic on mobile data.

### DL-17: Stored Credentials persisted for silent re-authentication
**Decision:** Username and password are stored on-device (DataStore + Android Keystore, `setUnlockedDeviceRequired(true)`) alongside the Refresh Token. Used silently when the Refresh Token cannot be renewed. A manual re-auth prompt is only shown if stored credentials also fail (e.g. password changed).
**Rationale:** Xexanos identified that truly silent re-auth after Refresh Token expiry requires stored credentials. Without them, an expired Refresh Token would always surface a login prompt - undermining the core promise of UJ-2.
**Security note:** No biometric gate added. `setUnlockedDeviceRequired(true)` ensures credentials are inaccessible while the device is locked. Appropriate threat model for a recipe/shopping list app.

### DL-19: Conflict resolution - updated_at last-write-wins (OQ-1 resolved)
**Decision:** Shopping List Items carry `created_at` and `updated_at` timestamps (confirmed via Mealie source). Conflict resolution uses last-write-wins on `updated_at`. No ETag or sequence field exists. Losing-side values logged locally in v1 to enable user-resolvable conflict surfacing post-v1.
**Rationale:** Most shopping list conflicts are benign (two people check the same item). Last-write-wins is sufficient for v1. User-resolvable conflict UI added to post-v1 backlog.

### DL-20: Single token model - no separate Refresh Token (OQ-2 resolved)
**Decision:** Mealie uses a single Access Token (JWT) with configurable lifetime via `TOKEN_TIME` environment variable. Default 48 hours; range 1-9,600 hours; differs per instance. No separate Refresh Token exists. The Stored Token is the on-disk copy of the most recently issued Access Token. All "Refresh Token" terminology removed from PRD.
**Rationale:** Confirmed via Mealie source code (`mealie/core/settings/settings.py`, `mealie/core/security/security.py`). `GET /api/auth/refresh` returns a new JWT using the existing token. Stored Credentials serve as fallback when the Stored Token is fully expired.

### DL-21: Single Household for v1, picker post-v1 (OQ-3 resolved)
**Decision:** V1 always uses `/api/households/self`. Multi-Household picker added to post-v1 backlog.
**Rationale:** Confirmed by Xexanos. Covers all known v1 users; complexity not warranted until the app has a broader audience.

### DL-22: APK for v1, F-Droid requirements for later iterations (OQ-4 resolved)
**Decision:** V1 distributed as direct APK. F-Droid and/or Play Store listing requires: privacy policy, compliant signing config, no closed-source SDKs. These requirements are documented in §6.2 for future iterations.
**Rationale:** Confirmed by Xexanos. V1 audience is Xexanos only; APK distribution is sufficient.

### DL-23: Monochrome adaptive icon - yes, AGPL-3.0 (OQ-5 resolved)
**Decision:** Monochrome adaptive icon included using a derivative of the Mealie icon. Mealie icon is AGPL-3.0 (no separate trademark or brand-protection policy). `android:monochromeIcon` supported from API 33 with graceful fallback on earlier levels.
**Rationale:** Confirmed by Xexanos; aligns with planned icon design. License verified via Mealie GitHub repository.

### DL-24: HTTP security warning - warn, allow, remember per URL (A1 resolved)
**Decision:** When the user enters an HTTP URL, the app shows a one-time security warning per server URL. The confirmation is persisted - the warning is not repeated on subsequent launches for the same URL.
**Rationale:** Self-hosted Mealie instances on local networks frequently run without TLS. Blocking HTTP entirely would break valid setups. Repeating the warning on every launch would be annoying for a deliberate choice. One-time confirmation per URL respects the Operator's decision while surfacing the risk at the right moment.

### DL-25: Offline Indicator placement deferred to UX design (A2 resolved)
**Decision:** Exact visual treatment of the Offline Indicator (chip, banner, icon) is a UX design decision. PRD specifies behaviour only: non-blocking, appears within 3 seconds of connectivity loss, disappears within 3 seconds of restoration.
**Rationale:** Placement and visual form belong in the UX design phase, not the PRD.
